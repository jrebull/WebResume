Servicios de nube: AWS vs GCP

Guía rápida para defender decisiones técnicas en entrevistas de system design. Usa esta hoja para mapear necesidades del sistema con servicios equivalentes en AWS y Google Cloud Platform.

Regla mental

Compute para correr lógica
Storage para archivos grandes
Database para datos operacionales
Queue para desacoplar servicios

Cómo contestar

No menciones solo el servicio. Explica qué problema resuelve: latencia, escala, costo, seguridad, resiliencia o simplicidad operacional.

Frase útil

“Separaría compute, storage, database, networking, messaging, security y observability para que cada capa escale de forma independiente.”

Tip de entrevista

Primero diseña de forma simple. Luego agrega cache, colas, CDN, réplicas, monitoreo y rate limiting según el cuello de botella.

Servicios que debes memorizar

Estos son suficientes para defender la mayoría de diseños de nivel entrevista.

Categoría AWS GCP Frase de defensa
Compute EC2 / ECS / Lambda Compute Engine / Cloud Run / Cloud Functions Uso compute para correr la lógica del backend.
Storage S3 Cloud Storage Uso object storage para archivos grandes y durables.
SQL RDS Cloud SQL Uso SQL cuando necesito transacciones y relaciones claras.
NoSQL DynamoDB Firestore Uso NoSQL para baja latencia y escala horizontal.
Cache ElastiCache Memorystore Uso cache para reducir latencia y lecturas repetidas.
Queue SQS Pub/Sub Uso colas para desacoplar servicios y absorber picos.
CDN CloudFront Cloud CDN Uso CDN para acercar contenido estático al usuario.
Monitoring CloudWatch Cloud Monitoring Uso monitoreo para detectar errores, latencia y saturación.

Ejemplo: API de documentos

Aplicado a POST, GET y PUT de documentos.

Versión AWS

  • API pública: API Gateway
  • Backend: ECS/Fargate, EKS o Lambda
  • Metadata: RDS o DynamoDB
  • Contenido: S3
  • Cache: ElastiCache
  • Procesamiento async: SQS
  • Observabilidad: CloudWatch

Versión GCP

  • API pública: API Gateway o Apigee
  • Backend: Cloud Run, GKE o Cloud Functions
  • Metadata: Cloud SQL o Firestore
  • Contenido: Cloud Storage
  • Cache: Memorystore
  • Procesamiento async: Pub/Sub
  • Observabilidad: Cloud Logging + Monitoring

“Para documentos grandes, no guardaría el contenido completo en la base de datos. Guardaría metadata en SQL o NoSQL y el archivo en object storage. Para tareas pesadas como indexar, extraer texto o generar thumbnails, usaría una cola para procesarlas de forma asíncrona.”
