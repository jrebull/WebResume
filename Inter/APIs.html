System Design Interview · API Playbook

Cómo usar APIs para defender cualquier diseño

Plantilla práctica para proponer endpoints, explicar decisiones y atacar casos tipo Spotify, TikTok, Google Docs, e-commerce, ride-sharing, YouTube, recommendation systems, DBMS, file system y link shortener.

Métodos que debes dominar

GET/api/resource/{id}
POST/api/resource
PUT/api/resource/{id}
PATCH/api/resource/{id}
DELETE/api/resource/{id}

1. Empieza por recursos

Identifica sustantivos: users, documents, products, rides, videos, recommendations, links.

2. Define operaciones

Crear, leer, actualizar, borrar, buscar, rankear, publicar, procesar o suscribirse.

3. Explica tradeoffs

Latencia, consistencia, escala, cache, colas, rate limit, seguridad y observabilidad.

4. Cierra con flujo

Cliente → API Gateway → Service → Cache/DB/Queue → Storage → Monitoring.

Fórmula para contestar APIs

Úsala cuando te pregunten “diseña X”. Te ayuda a sonar estructurado y senior.

Recurso¿Qué entidad central existe?
Endpoints¿Qué operaciones expongo?
Modelo¿Qué campos mínimos guardo?
Escala¿Cache, queue, shard, CDN?
SeguridadAuth, permisos, rate limits.

Cheat sheet de métodos HTTP

Esto debes defender con claridad. Es de lo primero que evalúan.

Método Uso Ejemplo Idempotencia Respuesta típica
GET Leer un recurso o lista de recursos. GET /api/documents/doc_123 Sí. No debe modificar estado. 200 OK, 404 Not Found
POST Crear recurso o disparar una acción. POST /api/documents No necesariamente. Puede crear duplicados. 201 Created, 202 Accepted
PUT Reemplazar o actualizar recurso completo. PUT /api/documents/doc_123 Sí. Repetirlo deja el mismo estado. 200 OK, 204 No Content
PATCH Actualizar parcialmente. PATCH /api/users/u_1/preferences Depende de la operación. 200 OK, 409 Conflict
DELETE Eliminar o marcar como eliminado. DELETE /api/links/abc123 Sí, si borrar dos veces no cambia el resultado. 204 No Content, 404 Not Found

Casos problema y APIs sugeridas

Ejemplos tipo entrevista. La clave no es memorizar, sino reconocer recursos y flujos.

1. E-commerce: top productos comprados

ClipKart.com

Mostrar top 10 productos más comprados en la última 1h, 4h o 24h.

  • Problema central: ranking en ventanas de tiempo.
  • Solución: eventos de compra → stream/queue → agregador → cache.
  • No hagas: calcular el top 10 escaneando toda la tabla de órdenes cada click.
GET /api/products/top?window=1h&limit=10
GET /api/products/top?window=4h&limit=10
GET /api/products/top?window=24h&limit=10
POST /api/orders

2. Ride-sharing service

Uber/Lyft

Clientes piden viaje; drivers cercanos reciben notificación en segundos.

  • Problema central: matching geoespacial en tiempo real.
  • Solución: location updates, geohash, pub/sub, WebSocket/push.
  • Riesgo: ubicación vieja o drivers duplicados aceptando el mismo ride.
POST /api/rides
PATCH /api/drivers/{driverId}/location
GET /api/drivers/nearby?lat=...&lng=...
POST /api/rides/{rideId}/accept

3. YouTube / streaming service

Video

Subir, procesar, almacenar y reproducir videos a gran escala.

  • Problema central: ingestión pesada y entrega de baja latencia.
  • Solución: object storage, transcoding async, CDN, metadata DB.
  • No hagas: servir videos directamente desde el app server.
POST /api/videos/upload-url
POST /api/videos/{videoId}/complete-upload
GET /api/videos/{videoId}
GET /api/videos/{videoId}/stream

4. Spotify

Music

Reproducir música, playlists, búsqueda y recomendaciones.

  • Problema central: catálogo, streaming, personalización.
  • Solución: metadata DB, object storage/CDN para audio, search index, reco service.
  • Cuidado: licencias, región, cache por usuario y derechos.
GET /api/tracks/{trackId}
GET /api/search?q=...
POST /api/playlists/{playlistId}/tracks
GET /api/recommendations?userId=...

5. TikTok feed

Feed ranking

Mostrar videos cortos personalizados con scroll infinito.

  • Problema central: ranking personalizado en baja latencia.
  • Solución: candidate generation, ranking model, feature store, cache.
  • Cuidado: cold start, diversidad, duplicados y freshness.
GET /api/feed?cursor=...
POST /api/events/watch
POST /api/events/like
GET /api/videos/{videoId}

6. Google Docs

Collaboration

Documentos editados por varios usuarios al mismo tiempo.

  • Problema central: concurrencia y colaboración en tiempo real.
  • Solución: operaciones incrementales, WebSocket, versioning, conflict resolution.
  • Cuidado: lost updates, permisos y audit log.
POST /api/documents
GET /api/documents/{documentId}
POST /api/documents/{documentId}/operations
GET /api/documents/{documentId}/changes?since=...

7. Recommendation Systems

Tu mero mole

Recomendar productos, videos, canciones o documentos relevantes.

  • Problema central: candidates + ranking + feedback loop.
  • Solución: eventos de usuario, feature store, modelo offline/online, cache.
  • Cuidado: latencia, sesgo, cold start y métricas.
GET /api/recommendations?userId=...&context=home
POST /api/events/impression
POST /api/events/click
POST /api/events/purchase

8. Link shortener

TinyURL/Bitly

Convertir URLs largas en links cortos y redirigir con baja latencia.

  • Problema central: generar IDs cortos únicos y redirigir rápido.
  • Solución: key generator, cache, DB key-value, analytics async.
  • Cuidado: colisiones, links maliciosos y rate limits.
POST /api/links
GET /{shortCode}
GET /api/links/{shortCode}/stats
DELETE /api/links/{shortCode}

9. File system

Dropbox/GDrive core

Crear carpetas, subir archivos, leer metadata, versionar y compartir.

  • Problema central: metadata tree + blobs grandes + permisos.
  • Solución: metadata DB, object storage, chunk upload, versioning.
  • Cuidado: rename/move atómico, permisos heredados y deduplicación.
POST /api/files/upload-url
POST /api/folders
GET /api/files/{fileId}/metadata
POST /api/files/{fileId}/share

10. DBMS / manejador de base de datos

Difícil pero defendible

Diseñar un sistema que recibe queries, guarda páginas, indexa y responde transacciones.

  • Problema central: storage engine, indexes, query planner, transactions.
  • Solución: B+ trees/LSM, buffer pool, WAL, locks/MVCC, replication.
  • Cuidado: ACID, recuperación ante fallas y consistencia.
POST /api/query
POST /api/transactions
POST /api/transactions/{txId}/commit
POST /api/transactions/{txId}/rollback

Qué mencionar para verte fuerte

Estas palabras elevan tu respuesta si las conectas con el problema correcto.

Tema Cuándo lo mencionas Frase de entrevista
Cache Lecturas frecuentes, rankings, perfiles, feed, metadata. “Cachearía respuestas calientes y pondría TTL para no servir datos eternamente viejos.”
Queue / PubSub Procesos pesados, emails, transcoding, analytics, indexing. “Lo haría asíncrono para desacoplar el request del procesamiento pesado.”
CDN Videos, imágenes, audio, archivos públicos o estáticos. “El app server no debe servir blobs grandes; uso object storage más CDN.”
Idempotency key Pagos, órdenes, creación de rides, retries del cliente. “Para evitar duplicados en retries, aceptaría una idempotency key por request.”
Rate limit APIs públicas, login, short links, feed, scraping. “Aplicaría rate limit por usuario, IP o API key para proteger disponibilidad.”
Versioning / ETag Docs, archivos, edición colaborativa, updates concurrentes. “Para evitar lost updates, usaría version o ETag con If-Match.”
Pagination / cursor Feeds, búsquedas, listas grandes, historial. “Usaría cursor pagination porque escala mejor que offset en datasets grandes.”
Observability Siempre. “Mediría latencia p50/p95/p99, tasa de errores, saturación y throughput.”

Script para abrir cualquier respuesta:
“Voy a empezar definiendo los recursos principales y las APIs. Después explico el flujo de escritura y lectura, dónde usaría cache o colas, cómo manejaría consistencia, seguridad, rate limits y observabilidad. Finalmente, reviso los tradeoffs y los posibles bottlenecks.”

Script para cerrar fuerte:
“El diseño inicial funciona para escala moderada. Para escalarlo, separaría la ruta crítica de los procesos asíncronos, agregaría cache para lecturas calientes, particionaría por claves naturales como userId, productId o region, y mediría p95/p99 para detectar cuellos de botella.”
