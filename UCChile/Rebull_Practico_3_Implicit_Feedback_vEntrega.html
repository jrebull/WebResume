Objetivo del Prctico

En este prctico implementaremos y compararemos sistemas de recomendacin basados en feedback implcito utilizando el dataset MovieLens:

  • Alternating Least Squares (ALS): Optimizacin mediante factorizacin matricial
  • Bayesian Personalized Ranking (BPR): Optimizacin basada en pares de preferencias
  • Evaluacin mediante mtricas de ranking: MAP y NDCG@10

Utilizaremos la librera implicit (https://github.com/benfred/implicit)

Introduccin

El feedback implcito representa seales indirectas de preferencia del usuario (visualizaciones, clicks, compras) a diferencia del feedback explcito (ratings). Los mtodos ALS y BPR son algoritmos especializados en este tipo de datos:

  • ALS factoriza la matriz usuario-item minimizando el error de reconstruccin con regularizacin.
  • BPR optimiza el ranking relativo entre items observados y no observados mediante una funcin objetivo bayesiana.

Este prctico explora cmo ambos mtodos se comportan al variar su hiperparmetro ms crtico: el nmero de factores latentes.

Carga de Dataset de Testing

Nota Metodolgica: Correcciones Crticas en la Carga de Datos (Train y Test)

Durante la revisin del cdigo, se detectaron dos reas de oportunidad metodolgicas en las celdas de carga de datos. La combinacin de estos errores afectaba la validez del entrenamiento y la evaluacin.

Problema 1: Error de Lgica (Filtro de Relevancia Faltante)

El ground truth (los datos contra los que evaluamos) y los datos de entrenamiento deben contener nicamente interacciones positivas (es decir, pelculas que al usuario s le gustaron, con rating >= 3, que se mape a 1).

El cdigo original omita este filtro (if row[3] == 1:).

  • Impacto en el Set de Entrenamient: El modelo estaba siendo entrenado con interacciones rating=0 (pelculas odiadas) como si fueran interacciones positivas, "ensuciando" los factores latentes.
  • Impacto en el Set de Prueba: El diccionario user_items_test (el ground truth) inclua tems que el usuario no consider relevantes. Esto penalizaba injustamente al modelo por no recomendar pelculas que el usuario odiaba.

Problema 2: Error de Indentacin (Ground Truth Incompleto)

Independiente del error anterior, el cdigo original para el set de prueba (Celda 8) tena la lnea .append() dentro del bloque if:

# CDIGO INCORRECTO
for row in df_test.itertuples():
  if row[1] not in user_items_test:
    user_items_test[row[1]] = []
    user_items_test[row[1]].append(row[2]) # <--- Error de indentacin

Consecuencia: Este error causaba que solo el primer tem de prueba de cada usuario fuera agregado a la lista. Esto haca que el ground truth estuviera incompleto (casi siempre con un solo tem).

La Correccin (Implementada)

Se aplicaron ambas correcciones en las celdas correspondientes. El cdigo ahora filtra correctamente por relevancia (if row[3] == 1:) y la indentacin es correcta, asegurando que se capturen todos los tems relevantes.

# CDIGO CORREGIDO
for row in df_test.itertuples():
  if row[3] == 1: # <-- FIX 1: Solo tems relevantes
    if row[1] not in user_items_test:
      user_items_test[row[1]] = []
    user_items_test[row[1]].append(row[2]) # <-- FIX 2: Indentacin corregida

# CDIGO CORREGIDO
for row in df_train.itertuples():
  if row[3] == 1: # <-- FIX 1: Solo tems relevantes
    if row[1] not in user_items:
      user_items[row[1]] = []
    # ... resto del cdigo ...

Impacto en el Prctico

La combinacin de estos dos errores era la causa de que los valores de MAP y NDCG@10 fueran idnticos en la ejecucin original. Al evaluar contra un ground truth de un solo tem (por el error de indentacin), las mtricas de ranking colapsan a una simple "Tasa de Acierto" (Hit Rate).

Con estas correcciones, el modelo se entrena solo con datos positivos y se evala contra un ground truth completo y relevante. Esto asegura que los valores de MAP y NDCG@10 sean significativos, distintos y reflejen una evaluacin real de la calidad del ranking del modelo. Todos los anlisis de sensibilidad y comparaciones en este prctico se basan en estas correcciones.

Definicin de Mtricas de Evaluacin

Las mtricas implementadas son:

  • Precision@K: Proporcin de items relevantes en top-K
  • MAP (Mean Average Precision): Promedio de la precisin en cada posicin relevante
  • NDCG@K (Normalized Discounted Cumulative Gain): Mtrica de ranking que penaliza posiciones bajas

Fuente: https://gist.github.com/bwhite/3726239

def precision_at_k(r, k):
    assert k >= 1
    r = np.asarray(r)[:k] != 0
    if r.size != k:
        raise ValueError('Relevance score length < k')
    return np.mean(r)

def average_precision(r):
    r = np.asarray(r) != 0
    out = [precision_at_k(r, k + 1) for k in range(r.size) if r[k]]
    if not out:
        return 0.
    return np.mean(out)

def mean_average_precision(rs):
    return np.mean([average_precision(r) for r in rs])

def dcg_at_k(r, k):
    r = np.asarray(r)[:k]
    if r.size:
        return np.sum(np.subtract(np.power(2, r), 1) / np.log2(np.arange(2, r.size + 2)))
    return 0.

def ndcg_at_k(r, k):
    idcg = dcg_at_k(sorted(r, reverse=True), k)

    if not idcg:
        return 0.
    return dcg_at_k(r, k) / idcg

Actualizacin Metodolgica: Mejora de la Funcin evaluate_model

Tras una revisin ms profunda del pipeline de evaluacin, se identificaron varias reas de oportunidad clave en la implementacin original de la funcin evaluate_model (definida originalmente). Estos puntos afectaban la precisin de los resultados numricos.

Puntos de Mejora en la Versin Original

La funcin inicial tena 5 debilidades metodolgicas que sesgaban las mtricas:

  1. Confusin de Dominios de ID (Usuario): Se iteraba usando el ID "raw" del usuario del set de prueba (ej. u = 943), pero se usaba este ID para consultar la matriz y el modelo (model.recommend(u, ...)), que esperan un ndice interno (ej. 800). Se estaban evaluando las recomendaciones de usuarios incorrectos.
  2. Fuga de Datos (Data Leakage): No se estaba utilizando filter_already_liked_items=True. El modelo poda obtener una puntuacin artificialmente alta por "recomendar" tems que el usuario ya haba visto en el set de entrenamiento.
  3. Confusin de Dominios de ID (tem): Se comparaban los ndices internos de los tems recomendados (devueltos por rec) con los IDs "raw" del ground truth (user_items_test[u]). Eran dominios incomparables.
  4. Vector de Relevancia Incorrecto: La lgica np.isin(ground_truth, rec) estaba invertida. La forma correcta es np.isin(rec, ground_truth) para crear un vector de longitud N en el orden de las recomendaciones.
  5. Promedio Errneo: Las mtricas se dividan por len(user_items_test) en lugar de por un contador de usuarios vlidos (aquellos que existen tanto en el set de train como en el de test).

Lgica de la Nueva Funcin Mejorada

La nueva funcin evaluate_model siguiente implementa estas mejoras:

  • 1. Mapeo Correcto de Usuario: Traduce el u_raw (ID raw) a su ndice interno u usando el diccionario user_ids antes de llamar a model.recommend.
  • 2. Sin Fuga de Datos: Activa filter_already_liked_items=True en la llamada al modelo.
  • 3. Mapeo Correcto de tem: Convierte los ndices internos de los tems recomendados (rec_idx) de vuelta a IDs "raw" (rec_raw) usando el array itemset.
  • 4. Vector Vlido: Compara rec_raw (recomendaciones) contra rel_raw (ground truth) en el orden correcto.
  • 5. Promedio Robusto: Utiliza un contador m de usuarios vlidos para la divisin final.

Impacto en el Entregable

Esta correccin es fundamental para la precisin del anlisis. Todos los valores de MAP y NDCG generados en los experimentos (base, sensibilidad ALS y sensibilidad BPR) han sido recalculados. Las conclusiones del anlisis ahora se basan en mtricas numricamente correctas y metodolgicamente vlidas, haciendo que la comparacin de "factores ptimos" y el anlisis general entre ALS y BPR sean fiables.

def evaluate_model(model, n):
    """
    Funcin de evaluacin robusta que maneja correctamente:
    1. Traduccin de IDs (raw vs. internos) para usuarios e tems.
    2. Filtrado de usuarios no vlidos (no en train o sin ground truth).
    3. Prevencin de fuga de datos (filter_already_liked_items=True).
    4. Creacin correcta del vector de relevancia.
    5. Promedio seguro con contador de usuarios vlidos (m).
    """
    mean_map = 0.0
    mean_ndcg = 0.0
    m = 0  # usuarios vlidos (presentes en train y con relevantes en test)

    # Iteramos sobre los usuarios (u_raw) y sus items relevantes (rel_raw) del test set
    for u_raw, rel_raw in user_items_test.items():

        # 1. Filtro de usuarios:
        # Si el usuario raw no est en el set de train (user_ids) O
        # si no tiene items relevantes en el test (not rel_raw), lo saltamos.
        if u_raw not in user_ids or not rel_raw:
            continue

        # 2. Traduccin de ID de Usuario:
        # Obtenemos el ndice INTERNO (u) a partir del ID RAW (u_raw)
        u = user_ids[u_raw]

        # 3. Obtener Recomendaciones:
        # Usamos el ndice INTERNO (u) y filtramos lo que ya vio en TRAIN
        rec_idx, _ = model.recommend(
            userid=u,
            user_items=user_item_matrix[u],
            N=n,
            filter_already_liked_items=True  # Evita fuga de datos
        )

        # 4. Traduccin de ID de tems:
        # Convertimos los ndices INTERNOS de tems (rec_idx) a IDs RAW (rec_raw)
        # usando el array 'itemset' (creado en Celda 11)
        rec_raw = itemset[rec_idx]

        # 5. Vector de Relevancia:
        # Comparamos los IDs RAW recomendados (rec_raw) vs los IDs RAW del ground truth (rel_raw)
        # El vector 'r' tendr longitud 'n' y estar en el orden de las recomendaciones.
        r = np.isin(rec_raw, rel_raw, assume_unique=False).astype(int)

        # 6. Sumar mtricas
        rel_vector = [r] # mean_average_precision espera una lista de listas
        mean_map  += mean_average_precision(rel_vector)
        mean_ndcg += ndcg_at_k(r, n) # ndcg_at_k espera el vector 'r'
        m += 1 # Contamos al usuario como vlido

    # 7. Promedio Seguro
    if m == 0:
        return 0.0, 0.0

    mean_map  /= m
    mean_ndcg /= m
    return mean_map, mean_ndcg

Anlisis de los Resultados

Aunque tericamente BPR est diseado para optimizar ranking directamente, en este caso ALS muestra mejor desempeo debido a:

1. Ventaja de Factorizacin Completa

  • ALS factoriza toda la matriz de interacciones simultneamente.
  • Captura relaciones globales ms efectivamente con pocos factores (10).

2. Convergencia Ms Estable

  • ALS utiliza optimizaciones cerradas (closed-form).
  • BPR emplea muestreo estocstico que requiere ms iteraciones para converger.

3. Sensibilidad a Hiperparmetros

  • BPR es ms sensible a learning rate, regularizacin y nmero de iteraciones.
  • La configuracin por defecto podra no ser ptima para este dataset.

4. Dimensionalidad Limitada

  • Con solo 10 factores latentes, la optimizacin directa de ALS es ms efectiva.
  • El muestreo por pares de BPR necesita ms dimensiones para capturar patrones complejos.

Conclusin

En configuraciones de baja dimensionalidad (10 factores), ALS puede superar a BPR. El anlisis de sensibilidad con ms factores latentes revelar si esta tendencia se mantiene o se invierte al incrementar la capacidad representacional de los modelos.

Pregunta 4: Interpretacin de Resultados (2 puntos)

Por qu los resultados no hacen mucho sentido? Explique la intuicin.

RESPUESTA:

Los resultados de pelculas similares a Star Wars (1977) pueden no tener sentido temtico porque BPR aprende patrones de comportamiento de usuarios, no caractersticas de contenido.

Qu significa "similaridad" en BPR?

NO significa: "Pelculas con temtica similar (sci-fi, space opera, etc.)"

S significa: "Pelculas vistas por usuarios con gustos similares"

Tres razones principales:

1. Aprende de co-ocurrencias, no de contenido

  • BPR analiza: "Usuarios que vieron Star Wars tambin vieron X"
  • NO analiza: Gnero, director, trama, actores

Ejemplo:

  • Si muchos usuarios ven Star Wars + Toy Story + Indiana Jones
  • BPR aprende que estas pelculas son "similares" (mismo pblico)
  • Aunque temticamente sean muy diferentes

2. Sesgo de popularidad

Las pelculas muy populares aparecen como "similares" a casi todo:

  • Si 70% de usuarios vio Star Wars
  • Y 60% de usuarios vio Pelcula X
  • Hay alto overlap estadstico BPR los considera similares

3. Factores latentes capturan comportamiento, no atributos

Los embeddings NO representan:

  • "Pelcula de ciencia ficcin"
  • "Tiene batallas espaciales"
  • "pica de los 80s"

Los embeddings S representan:

  • "Perfil de usuario que la consume"
  • "Patrn de visualizacin agregado"
  • "Correlaciones implcitas en preferencias"

Es esto un problema?

NO. Es el comportamiento esperado de Collaborative Filtering.

Ventaja: Descubre relaciones no obvias

  • "Fans de Star Wars tambin disfrutan pelculas X" (que no esperaras por gnero)
  • til para recomendaciones personalizadas

Limitacin: No sirve para "buscar pelculas parecidas temticamente"

Soluciones si necesitas similaridad de contenido:

Enfoque Qu usa Resultado
Content-Based Gneros, directores, plot Pelculas temticamente similares
Collaborative (BPR) Comportamiento de usuarios Preferencias correlacionadas
Hbrido Ambos Mejor de ambos mundos

Conclusin

BPR optimiza para predecir qu le gustar al usuario, no para encontrar pelculas similares en contenido.

Son objetivos diferentes:

  • Recomendacin personalizada Usa Collaborative Filtering
  • "Ms pelculas como esta" Usa Content-Based

Los resultados "no tienen sentido temtico" pero s tienen sentido para recomendacin, que es el propsito del modelo.

Conclusiones y Reflexin Personal

Conclusiones Principales

  1. ALS supera a BPR en este experimento: Contrario a la expectativa terica, ALS (con 10 factores), en su configuracin ms simple, obtuvo un rendimiento (MAP = 0.5913) significativamente superior al de BPR (con 150 factores) en su punto ptimo (MAP = 0.4840).

  2. "Menos es Ms" (Para ALS): El punto ptimo para ALS fue 10 factores. El anlisis de sensibilidad demostr un caso claro de sobreajuste (overfitting) inmediato: cada factor adicional degrad el rendimiento, colapsando en un 74% con 500 factores.

  3. Comportamientos Opuestos ante la Complejidad: Los dos modelos reaccionaron de forma opuesta al aumento de factores.

    • ALS sufri de sobreajuste: su rendimiento decay con ms factores.
    • BPR sufri de subajuste (underfitting): su rendimiento mejor con ms factores hasta alcanzar un plateau (meseta) estable alrededor de los 150-300 factores.

  4. La Configuracin por Defecto Gan (Para ALS): La configuracin base de implicit para ALS (10 factores) result ser la ptima. En contraste, BPR requiri un anlisis de sensibilidad para encontrar su punto ptimo, demostrando que ALS fue ms robusto "fuera de la caja" para este dataset.
